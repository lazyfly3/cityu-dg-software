## 1. Software Engineering Overview / 1. 软件工程概述

### What is Software Engineering? / 什么是软件工程？

#### English

Software Engineering is the art of creation, collaboration, and solving complex problems. It involves systematic, standardized, and quantifiable methods to develop large-scale software systems. Unlike personal programming projects that rely on intuition and experience, software engineering requires detailed blueprints, strict processes, and continuous maintenance.

#### 中文

软件工程是创造、协作和解决复杂问题的艺术。它涉及系统化、标准化和可量化的方法来开发大规模软件系统。与依赖直觉和经验个人编程项目不同，软件工程需要详细的蓝图、严格的流程和持续的维护。

---

### Programming vs. Software Engineering / 编程 vs. 软件工程

#### English

**Personal Projects (Programming)**:
- Relies on intuition and experience
- Simple tools, individual effort
- Goal: Rapid feature implementation, minimum viable solution

**Large-Scale Software (Software Engineering)**:
- Systematic, standardized, and quantifiable methods
- Detailed blueprints, strict processes, and continuous maintenance
- Goal: Scale (handle massive amounts of data), Complexity (manage intricate systems), Collaboration (enable coordination among large teams), Quality (ensure reliability, security, and performance)

#### 中文

**个人项目（编程）**：
- 依赖直觉和经验
- 简单工具，个人努力
- 目标：快速实现功能，最小可行方案

**大规模软件（软件工程）**：
- 系统化、标准化和可量化的方法
- 详细的蓝图、严格的流程和持续的维护
- 目标：规模（处理大量数据）、复杂性（管理复杂系统）、协作（实现大型团队的协调）、质量（确保可靠性、安全性和性能）

### Modern Development Pattern / 现代开发模式

#### English

**Traditional Development Pattern**: Step-by-step progression; only see the final product after long period. High risks.

**Modern Development Pattern**: Done is better than perfect. First, build a usable core product (MVP), then continuously iterate and improve.
- Small Rapid Iterations
- Quick Feedback
- Embracing Change

#### 中文

**传统开发模式**：逐步推进；只有在长时间后才能看到最终产品。高风险。

**现代开发模式**：完成比完美更好。首先，构建一个可用的核心产品（MVP），然后持续迭代和改进。
- 小规模快速迭代
- 快速反馈
- 拥抱变化

## 2. Course Project / 2. 课程项目

### Project Goal / 项目目标

#### English

This is a code project and a team-based project. The goal is to develop a software engineering tool, in which you realize your original new idea. A software engineering tool is for software developers to assist the development team in carrying out their development technical tasks. It is not developing software for ordinary people (e.g., don't develop apps we install on our mobile phones for daily use).

#### 中文

这是一个代码项目和基于团队的项目。目标是开发一个软件工程工具，在其中实现你的原创新想法。软件工程工具是为软件开发者服务的，用于协助开发团队完成其开发技术任务。它不是为普通用户开发软件（例如，不要开发我们在手机上日常使用的应用程序）。

### Project Size / 项目规模

#### English

The project accounts for 25% of the coursework component. For a team of 6 students, the project size is akin to a single-person final-year project (FYP) in your undergraduate degree. Assessment Expectation: Each student will spend 30 to 40 hours on the project.

#### 中文

项目占课程作业部分的 25%。对于 6 名学生的团队，项目规模类似于本科学位的单人毕业设计（FYP）。评估期望：每个学生将在项目上花费 30 到 40 小时。

### Project Progress / 项目进度

#### English

**Team size**: 5-8 students
- 5-6 students: at least two non-functional requirements
- 7-8 students: at least three non-functional requirements

Each non-functional requirement is required to be addressed by a software architectural design option with rationale. Apply an Agile methodology with at least three cycles (sprints in Scrum).

**Sprint 1 (Weeks 2-5)**: Develop user stories, sketch architecture to address non-functional requirements, code, test, review, identify/track/resolve technical debts

**Sprint 2 (Weeks 6-9)**: Activities in Sprint 1 + code quality

**Sprint 3 (Weeks 10-13)**: Activities in Sprint 2 + automated testing, debugging and maintenance support, project wrap up, presentation, video, report writing

Each team must maintain a project in a code repository (e.g., Github) which captures the team's weekly activities to serve as evidences of activities conducted during the project period stated in the final report.

#### 中文

**团队规模**：5-8 名学生
- 5-6 名学生：至少两个非功能性需求
- 7-8 名学生：至少三个非功能性需求

每个非功能性需求都需要通过软件架构设计选项及其理由来解决。应用敏捷方法，至少三个周期（Scrum 中的冲刺）。

**冲刺 1（第 2-5 周）**：开发用户故事，绘制架构以解决非功能性需求，编码、测试、审查、识别/跟踪/解决技术债务

**冲刺 2（第 6-9 周）**：冲刺 1 的活动 + 代码质量

**冲刺 3（第 10-13 周）**：冲刺 2 的活动 + 自动化测试、调试和维护支持，项目收尾、演示、视频、报告撰写

每个团队必须在代码仓库（如 Github）中维护一个项目，记录团队的每周活动，作为最终报告中所述项目期间进行的活动的证据。

### Software Engineering Tool / 软件工程工具

#### English

A software engineering tool is a tool to help software developers (programmers or project teams) to complete software development tasks. It is not operating a software after software/service deployment.

Examples: a scrum tool, project management tool, a code review assistance, a testing tool, a debugging tool, a code smell tool, a technical debt tool, a design framework, a requirement tracking tool, a tool to identify critical issue in app review pool, etc.

It can be developed using whatever programming languages you choose. It can be developed as a standalone tool, a web service, an app, a browser extension, plugin, etc.

**Counterexamples** (Don't do it):
- Developing a web-based e-commerce system for selling and buying items
- Developing a library loan system
- Developing a machine learning model
- Developing a script to train a machine learning model

#### 中文

软件工程工具是帮助软件开发者（程序员或项目团队）完成软件开发任务的工具。它不是软件/服务部署后操作软件。

示例：scrum 工具、项目管理工具、代码审查辅助工具、测试工具、调试工具、代码异味工具、技术债务工具、设计框架、需求跟踪工具、识别应用审查池中关键问题的工具等。

它可以使用您选择的任何编程语言开发。它可以开发为独立工具、Web 服务、应用程序、浏览器扩展、插件等。

**反例**（不要这样做）：
- 开发用于买卖商品的基于 Web 的电子商务系统
- 开发图书馆借阅系统
- 开发机器学习模型
- 开发用于训练机器学习模型的脚本

### Project Recommendations / 项目建议

#### English

**Warning**: A full-feature tool is too large for a student team to target it within the project timeline (12 weeks). Don't do it!

**Recommendation**: Start with an operable open-source project (e.g., a project with its source code you can get from Github) or a project allows plug-in development. The best kind is a tool related to a research paper so that you can read the paper about the key algorithm implemented in the tool. Implement your original/improved algorithms to solve a specific software development problem. A tool can be a standalone tool or a component of a larger tool (e.g., a plugin in Eclipse IDE).

**Some Areas You May Consider**:
- Improvement on existing tasks: Develop an alternate method to handle the task
- Lower the acceptance boundary of an existing tool: Develop a module to solve setup or assumption problems
- Extension on top of existing tool: Modify or enhance the tool to solve a related problem
- Changing the applicability of an existing tool: Change static analysis component to dynamic analysis component, and vice versa
- Your Own Idea: Highly preferable. Do not bound by what have been described

#### 中文

**警告**：对于学生团队来说，完整功能的工具太大，无法在项目时间表（12 周）内完成。不要这样做！

**建议**：从一个可操作的开放源代码项目开始（例如，可以从 Github 获取源代码的项目）或允许插件开发的项目。最好的类型是与研究论文相关的工具，这样您可以阅读有关工具中实现的关键算法的论文。实现您的原创/改进算法来解决特定的软件开发问题。工具可以是独立工具或更大工具的组件（例如，Eclipse IDE 中的插件）。

**您可以考虑的一些领域**：
- 改进现有任务：开发处理任务的替代方法
- 降低现有工具的接受边界：开发模块来解决设置或假设问题
- 在现有工具之上扩展：修改或增强工具以解决相关问题
- 改变现有工具的适用性：将静态分析组件更改为动态分析组件，反之亦然
- 您自己的想法：非常可取。不要受已描述内容的限制
